## Why

ClickHouse SQL accepts `#` as a single-line comment terminator, identical in scoping behaviour to `--`. Real-world queries — particularly those generated by dashboard tools that mix shell-style and SQL-style commentary — rely on `#` working the same as `--`. Today the lexer recognises only `--` and `/* … */`, so any input containing a `#` comment fails to lex and the query never reaches the parser.

This change is a single-line lexer extension that closes the gap without touching the parser, the AST, the formatter, or any existing test fixture.

## What Changes

- The lexer's comment-skipping path recognises `#` as the start of a single-line comment. Everything from `#` through the next newline (or end-of-input) is treated as a comment and skipped, identical to the existing `--` behaviour.
- The remaining comment forms — `--` single-line and `/* … */` multi-line — are unchanged.
- No new token kind, no AST shape change, no parser-level work, no formatter change.

## Capabilities

### New Capabilities
- `hash-line-comments`: Recognise `#` as a single-line comment terminator in the SQL lexer, with the same scoping rules as `--`.

### Modified Capabilities
<!-- None. -->

## Impact

- **Code touched**: one new switch arm in `func (l *Lexer) skipComments()` in `parser/lexer.go`. The existing `consumeSingleLineComment` helper is reused.
- **Behavioural contract**: a lexer-level test (`TestConsumeHashComment` in `parser/lexer_test.go`) exercises leading, trailing, and consecutive `#` comments, and is the gating signal for the implementation. The existing `TestConsumeComment` covering `--` and `/* */` is a regression guard that must remain green.
- **No `.sql` fixture under `parser/testdata/` uses `#` comments today**, so existing goldens are expected to remain byte-identical. Any drift would indicate an unintended rendering change.
- **No dependencies** added, no public API change, no breaking changes.
